# Vault 폴더 구성 가이드

> "어디에 저장하느냐"가 정보를 살리느냐 죽이느냐를 결정한다.
> 각 폴더는 역할이 다르다. 폴더 = 정보의 생애 단계.

---

## 전체 구조 한눈에

```
vault/brain/
├── 00_inbox/          ← 들어오는 모든 것. 아직 처리 안 된 것
├── 10_daily/          ← 오늘 하루 CEO 노트
├── 20_depts/          ← 부서별 정제 자산 (실무 지식)
│   ├── finance/
│   ├── dev/
│   ├── hr/
│   ├── it/
│   ├── marketing/
│   ├── purchasing/
│   └── strategy/
├── 30_cross_functional/  ← 부서 경계를 넘는 프로젝트
├── 40_resources/         ← 학습·스킬·프롬프트 라이브러리
├── 50_personal/          ← 개인 영역 (목장, 투자, 일상)
├── 70_decisions/         ← 의사결정 기록
└── 90_meta/              ← 시스템 자체 문서
```

---

## 00_inbox/ — 들어오는 모든 것

### 역할
**처리 전 대기실.** 판단 없이 일단 모두 넣는다.
Outlook 메일, Claude Code 대화, 웹 클리핑, 드롭 파일 — 형태 불문 모든 인풋이 여기 쌓인다.
여기 있는 노트는 아직 "살아있는 정보"가 아니다. 처리(triage·distill)를 통해 진짜 자산이 된다.

### 하위 채널
| 채널 | 소스 |
|---|---|
| `00_inbox/claude/` | Claude Code 대화 세션 자동 저장 |
| `00_inbox/web/` | 웹 클리핑·뉴스 |
| `00_inbox/books/` | 독서 캡처 |
| `00_inbox/_processed/` | 처리 완료 보관 |

### Ethan이 쓰는 프롬프트
```
"inbox 트리아지 해줘. 오늘 들어온 것들 분류하고
 버릴 것·보관할 것·정제할 것 나눠줘."

"이번 주 inbox 전부 distill 후보 뽑아줘."
```

### 팀원이 확장할 수 있는 부분
> 팀원 각자가 **자기 inbox 채널** 하나씩 운영하면 된다.
> 예: `00_inbox/hana_kim/` — 팀원이 캡처한 메일·메모 대기

```
[팀원 프롬프트 예시]
"내가 오늘 저장한 메일 3개를 inbox에 넣고,
 어떤 게 Finance팀에 공유할 만한지 체크해줘."
```

## 10_daily/ — 오늘 하루 CEO 노트

### 역할
**하루의 기록.** 브리핑, 결정, 메모, 오늘 처리한 일의 흔적.
`daily-brief` 스킬이 매일 아침 자동으로 `2026-06-24_브리핑A.md` 형태로 생성한다.

### Ethan이 쓰는 프롬프트
```
"오늘 브리핑A 만들어줘."
→ 외부 뉴스 3줄 + 현안 키워드 + 오늘 할 일 자동 생성

"오늘 미팅에서 결정한 것들 10_daily에 기록해줘.
 날짜: 2026-06-24, 주제: iLumen GFM 보고."
```

### 팀원이 확장할 수 있는 부분
```
[팀원 프롬프트 예시]
"오늘 내가 처리한 업무 3건을 일일 노트로 정리해줘.
 형식: 날짜 / 업무명 / 완료여부 / 다음 액션."
```

## 20_depts/ — 부서별 정제 자산

### 역할
**실무 지식의 정본.** inbox에서 처리된 정보가 부서별로 여기 정착한다.
여기 있는 파일은 "검증·정제된 것"이다. 삭제하지 않고, 수정할 때는 날짜와 이유를 남긴다.

### 부서별 예시 자산

| 부서 | 자산 예시 |
|---|---|
| `finance/` | 자금계획.xlsx, 세금계산서 분석.md, 회계처리 메모 |
| `dev/` | 자동화 스크립트, bank_dl.py, 파이프라인 설계 |
| `hr/` | 취업규칙 검토, 급여 관련 메모 |
| `it/` | 쿠팡이츠 AML 분석, 시스템 이슈 |
| `marketing/` | 프로모션 분석, 배달앱 데이터 |
| `purchasing/` | 발주 이력, 계약 검토 |
| `strategy/` | PHK 매각 현황, 경쟁사 분석 |

### Ethan이 쓰는 프롬프트
```
"더존 세금계산서 집계누락 원인 분석 결과를
 finance 노트로 정리해줘."

"bank_dl.py 실행 결과 CSV를
 finance/자금계획에 반영해줘."

"오늘 확인한 Yum 청구 불일치 건을
 finance 노트로 기록해줘. 삼각검증 항목 포함."
```

### 팀원이 확장할 수 있는 부분
```
[팀원 프롬프트 예시]
"내가 담당하는 [구매팀] 업무 중
 반복적으로 쓰는 정보를 purchasing/ 폴더에
 노트로 정리해줘. 형식: 업무명·절차·담당자."

"이번 달 마케팅 프로모션 결과를
 marketing/2026-06_프로모션-결과.md 로 저장해줘."
```

## 30_cross_functional/ — 부서 경계를 넘는 프로젝트

### 역할
**여러 부서가 함께 만드는 자산.** Finance+IT, Finance+HR처럼 단일 부서로 귀속되지 않는 프로젝트가 여기 산다.
현재: `PHK-AI교육/` — Finance팀 전체가 사용자인 프로젝트.

### 폴더 구조 예시
```
30_cross_functional/
└── PHK-AI교육/
    ├── _project.md              ← 프로젝트 목적·현황·담당자
    ├── 2026-06-23_프롬프트카드.md
    ├── 2026-06-24_아웃룩-ClaudeCode-활용범위.md
    └── (이 파일)
```

### Ethan이 쓰는 프롬프트
```
"PHK-AI교육 Week 1 발표 자료 초안 만들어줘.
 대상: Finance팀 7명. 시간: 10분."

"PHK-AI교육 프로젝트 현황 업데이트해줘.
 오늘 완료: 킥오프 발표 자료 확정."
```

### 팀원이 확장할 수 있는 부분
```
[팀원 프롬프트 예시]
"PHK-AI교육 중 내가 실제로 써본 프롬프트 3개를
 PHK-AI교육/팀원-경험.md에 추가해줘."

"팀에서 공유하고 싶은 엑셀 자동화 팁을
 cross_functional/Finance팀-AI팁.md에 정리해줘."
```

## 40_resources/ — 학습·스킬·프롬프트 라이브러리

### 역할
**재사용 가능한 지식 저장소.** 한 번 배운 것이 다음에도 써먹힐 수 있도록 정제·구조화해서 보관.

### 하위 구조
```
40_resources/
├── learning/        ← 논문·책·뉴스 정제 노트
│   └── books/       ← 완독 독서 노트 (Quartz 공개)
├── skills/          ← Claude Code 스킬 정의 (자동화 레시피)
└── prompts/         ← 자주 쓰는 프롬프트 모음
```

### Ethan이 쓰는 프롬프트
```
"이 논문 핵심만 learning 노트로 정리해줘.
 삼각검증 필요 항목 표시 포함."

"/knowledge-ingest [URL]
 → 외부 아티클 3단계 버킷으로 정제"
```

### 팀원이 확장할 수 있는 부분
```
[팀원 프롬프트 예시]
"내가 자주 쓰는 엑셀 VBA 코드를
 resources/prompts/excel-vba-팁.md로 저장해줘."

"이 HBR 아티클 읽었는데
 Finance팀에 쓸 수 있는 부분만 3줄로 요약해줘.
 resources/learning에 저장해줘."
```

## 50_personal/ — 개인 영역

### 역할
**업무와 분리된 개인 기록.** 목장, 개인 투자, 가족 관련 메모 등.
시스템에서 자동 스캔 제외. 다른 노트에서 인용·참조하지 않는다.

---

## 70_decisions/ — 의사결정 기록

### 역할
**"왜 그 결정을 했는가"의 정본.** 나중에 돌아봤을 때 맥락을 복원할 수 있게.
큰 결정(계약, 채용, 투자, 전략 방향)은 여기 기록한다.

### Ethan이 쓰는 프롬프트
```
"/decision
 주제: Yum 청구 방식 변경 승인
 배경: 현행 방식의 리스크
 선택지 3개: ...
 → 달리오 5단계 구조로 기록"
```

### 팀원이 확장할 수 있는 부분
```
[팀원 프롬프트 예시]
"오늘 팀장님과 논의한 결정 사항을
 간단한 의사결정 노트로 기록해줘.
 형식: 결정내용 / 이유 / 다음 액션."
```

---

## 90_meta/ — 시스템 자체 문서

### 역할
**vault가 vault 자신에 대해 아는 것.** 시스템 지도, 운영 리듬, HANDOFF, 원장.
일반적으로 직접 편집하지 않음 — 스크립트와 Claude가 자동 갱신.

---

## 요약: 폴더별 한 줄 정의

| 폴더 | 한 줄 정의 | 누가 주로 씀 |
|---|---|---|
| `00_inbox/` | 판단 전 대기실 | 자동 + 전체 |
| `10_daily/` | 오늘의 기록 | Ethan |
| `20_depts/` | 부서별 정제 정본 | Ethan + 팀원 확장 가능 |
| `30_cross_functional/` | 팀 공유 프로젝트 | Ethan + 팀원 기여 |
| `40_resources/` | 재사용 지식 라이브러리 | Ethan + 팀원 축적 |
| `50_personal/` | 개인 영역 | Ethan 전용 |
| `70_decisions/` | 의사결정 정본 | Ethan |
| `90_meta/` | 시스템 자체 | 자동 |

---

> **팀원 참여 진입점 요약**
> 처음 시작한다면: `20_depts/<내 부서>/` 또는 `30_cross_functional/PHK-AI교육/` 에서 시작.
> 내가 반복적으로 찾는 정보를 노트 하나로 만드는 것이 첫 번째 미션.
